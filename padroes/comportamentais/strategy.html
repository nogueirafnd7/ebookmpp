Escolhendo estratégias musicais

O que é o Padrão Strategy?

O Padrão Strategy é um padrão de projeto comportamental que permite definir uma família de algoritmos, encapsular cada um deles e torná-los intercambiáveis. O Strategy permite que o algoritmo varie independentemente dos clientes que o utilizam [1].

Este padrão é particularmente útil quando você tem várias maneiras de executar uma operação e deseja poder alternar entre elas em tempo de execução. Em vez de implementar todos os algoritmos em uma única classe com múltiplas condições, o Strategy separa cada algoritmo em sua própria classe.

Capa Próximo

O que ele resolve?

O principal problema que o Padrão Strategy resolve é a rigidez de algoritmos em sistemas que precisam alternar entre diferentes formas de executar uma mesma tarefa. Sem este padrão, o código frequentemente se torna uma série de declarações if-else ou switch-case que selecionam qual algoritmo usar.

Isso leva a um código difícil de manter, estender e testar. Além disso, qualquer mudança nos algoritmos ou adição de novos algoritmos requer modificar a classe original, violando o princípio Open/Closed. O Strategy Pattern resolve isso encapsulando cada algoritmo em uma classe separada, permitindo que novos algoritmos sejam adicionados sem modificar o contexto existente [2].

Diagrama de Exemplo Genérico (UML)

O diagrama de classes UML a seguir ilustra a estrutura fundamental do Padrão Strategy:

  • Context: Mantém uma referência para um objeto Strategy. Pode definir uma interface que permite ao Strategy acessar seus dados.
  • Strategy (Interface): Declara uma interface comum a todos os algoritmos suportados. O Context usa esta interface para chamar o algoritmo definido por um ConcreteStrategy.
  • ConcreteStrategy: Implementa o algoritmo usando a interface Strategy. Cada ConcreteStrategy fornece sua própria implementação do algoritmo.

O Context configura-se com um objeto ConcreteStrategy e delega a ele a execução do algoritmo. Os clientes normalmente criam e passam um objeto ConcreteStrategy para o Context.

Exemplo Fictício (Com Britney Spears)

Vamos imaginar a Britney Spears, uma artista que precisa escolher diferentes estratégias de performance para diferentes tipos de shows:

  • Estratégia Pop: Para shows de música pop com coreografias elaboradas e hits como "Baby One More Time"
  • Estratégia Acústica: Para performances intimistas com versões acústicas de suas músicas
  • Estratégia Vegas: Para seu residência em Las Vegas com produções grandiosas e efeitos especiais

Cada estratégia define uma maneira diferente de executar um show. O padrão Strategy permite que ela mude facilmente entre essas estratégias dependendo do contexto do evento.

Sem o Padrão: O que Acontece?

Sem o Padrão Strategy, o código da Britney Spears teria que verificar constantemente qual estratégia usar para cada show:


class BritneySpears:
    def perform_show(self, show_type):
        if show_type == "POP":
            self._setup_pop_show()
            self._execute_pop_choreography()
            self._sing_pop_hits()
        elif show_type == "ACOUSTIC":
            self._setup_acoustic_stage()
            self._play_guitar()
            self._sing_acoustic_versions()
        elif show_type == "VEGAS":
            self._setup_vegas_stage()
            self._execute_elaborate_choreography()
            self._use_special_effects()
    
    def _setup_pop_show(self):
        print("Preparando palco para show pop...")
    
    def _setup_acoustic_stage(self):
        print("Preparando palco acústico...")
    
    def _setup_vegas_stage(self):
        print("Preparando palco de Vegas...")
    
    # ... outros métodos específicos para cada tipo de show

Este código se torna difícil de manter à medida que novos tipos de shows são adicionados, violando o princípio Open/Closed.

Com o Padrão Aplicado (Pseudo Linguagem)

Aplicando o Padrão Strategy, encapsulamos cada estratégia em sua própria classe:


// Interface Strategy
interface PerformanceStrategy {
    void setupStage();
    void executePerformance();
}

// Estratégias Concretas
class PopStrategy implements PerformanceStrategy {
    public void setupStage() {
        print("Preparando palco para show pop...");
    }
    
    public void executePerformance() {
        print("Executando coreografia elaborada e cantando hits pop!");
    }
}

class AcousticStrategy implements PerformanceStrategy {
    public void setupStage() {
        print("Preparando palco acústico...");
    }
    
    public void executePerformance() {
        print("Tocando guitarra e cantando versões acústicas!");
    }
}

class VegasStrategy implements PerformanceStrategy {
    public void setupStage() {
        print("Preparando palco de Vegas com efeitos especiais...");
    }
    
    public void executePerformance() {
        print("Executando coreografia elaborada com efeitos especiais!");
    }
}

// Context
class BritneySpears {
    private PerformanceStrategy currentStrategy;
    
    public void setStrategy(PerformanceStrategy strategy) {
        this.currentStrategy = strategy;
    }
    
    public void performShow() {
        currentStrategy.setupStage();
        currentStrategy.executePerformance();
    }
}

// Cliente
public class Client {
    public static void main(String[] args) {
        BritneySpears britney = new BritneySpears();
        
        // Show pop
        britney.setStrategy(new PopStrategy());
        britney.performShow();
        
        // Show acústico
        britney.setStrategy(new AcousticStrategy());
        britney.performShow();
    }
}

Exemplo Prático do Dia a Dia (Python)

Um exemplo prático do Padrão Strategy é um sistema de processamento de pagamentos que pode usar diferentes métodos (cartão de crédito, PayPal, transferência bancária):


from abc import ABC, abstractmethod

class PaymentStrategy(ABC):
    @abstractmethod
    def process_payment(self, amount):
        pass

class CreditCardStrategy(PaymentStrategy):
    def __init__(self, card_number, expiry_date, cvv):
        self.card_number = card_number
        self.expiry_date = expiry_date
        self.cvv = cvv
    
    def process_payment(self, amount):
        print(f"Processando pagamento de R${amount} via cartão de crédito {self.card_number[-4:]}")
        # Lógica específica para cartão de crédito

class PayPalStrategy(PaymentStrategy):
    def __init__(self, email):
        self.email = email
    
    def process_payment(self, amount):
        print(f"Processando pagamento de R${amount} via PayPal ({self.email})")
        # Lógica específica para PayPal

class BankTransferStrategy(PaymentStrategy):
    def __init__(self, account_number, bank_code):
        self.account_number = account_number
        self.bank_code = bank_code
    
    def process_payment(self, amount):
        print(f"Processando pagamento de R${amount} via transferência bancária")
        # Lógica específica para transferência bancária

class PaymentProcessor:
    def __init__(self):
        self.strategy = None
    
    def set_payment_strategy(self, strategy):
        self.strategy = strategy
    
    def process(self, amount):
        if self.strategy:
            self.strategy.process_payment(amount)
        else:
            print("Nenhuma estratégia de pagamento definida!")

# Uso do sistema
processor = PaymentProcessor()

# Pagamento com cartão de crédito
processor.set_payment_strategy(CreditCardStrategy("1234567890123456", "12/25", "123"))
processor.process(150.00)

# Pagamento com PayPal
processor.set_payment_strategy(PayPalStrategy("britney@example.com"))
processor.process(200.00)

# Pagamento com transferência bancária
processor.set_payment_strategy(BankTransferStrategy("987654", "001"))
processor.process(300.00)

Vantagens

O Padrão Strategy oferece várias vantagens significativas:

  • Flexibilidade de Algoritmos: Permite trocar algoritmos em tempo de execução, oferecendo grande flexibilidade.
  • Elimina Condições Complexas: Remove a necessidade de longas declarações if-else ou switch-case que selecionam algoritmos.
  • Facilita a Adição de Novos Algoritmos: Novos algoritmos podem ser adicionados criando novas classes sem modificar o Context ou algoritmos existentes, seguindo o princípio Open/Closed.
  • Separação de Responsabilidades: Separa a implementação do algoritmo do contexto que o utiliza, promovendo um código mais limpo e organizado.
  • Reutilização de Algoritmos: Os algoritmos podem ser reutilizados em diferentes contextos, promovendo a reutilização de código.
Anterior Próximo

Desvantagens

O Padrão Strategy também apresenta algumas desvantagens:

  • Sobrecarga de Classes: Pode resultar em um grande número de classes pequenas, especialmente se há muitos algoritmos, o que pode aumentar a complexidade do sistema.
  • Complexidade para Algoritmos Simples: Para sistemas com poucos algoritmos ou algoritmos simples, o padrão pode ser excessivo e adicionar complexidade desnecessária.
  • Cliente Precisa Conhecer as Estratégias: O cliente precisa estar ciente das diferentes estratégias disponíveis para selecionar a apropriada.
  • Comunicação entre Strategy e Context: Pode haver overhead de comunicação se a Strategy precisa de muitos dados do Context.

Conclusão

O Padrão Strategy é uma ferramenta poderosa para gerenciar famílias de algoritmos intercambiáveis. Ele é particularmente útil quando você precisa selecionar entre diferentes formas de executar uma operação em tempo de execução.

Ao encapsular cada algoritmo em uma classe separada, o padrão promove um código mais limpo, organizado e fácil de manter. Ele elimina a necessidade de condicionais complexas e facilita a adição de novos algoritmos no futuro.

No entanto, como todo padrão de projeto, é importante avaliar se o Strategy Pattern é a solução adequada para o problema em questão. Para sistemas com poucos algoritmos ou algoritmos simples, uma implementação mais direta pode ser mais apropriada.
